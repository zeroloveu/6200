# `app/main.py` 模块说明

## 1. 模块定位

`app/main.py` 是整个 Web 应用的核心控制层，也是 FastAPI 的主入口。  
它负责把前端页面、数据库模型、链上服务和用户会话整合起来，形成完整的业务流程。

简而言之，这个模块解决的是“用户在网页上的每一步操作应该如何被系统接收、校验、处理和反馈”。

## 2. 主要职责

- 创建 FastAPI 应用实例并注册生命周期函数
- 挂载静态资源目录与模板目录
- 管理会话登录状态
- 提供首页、注册、登录、个人资料、投票创建、投票详情、结果查看等路由
- 处理投票创建、编辑、删除、同步链上交易等核心业务逻辑
- 负责把数据库中的投票数据和链上真实状态组合后返回到页面

## 3. 核心功能拆解

### 3.1 页面与模板渲染

模块通过 `Jinja2Templates` 将后端数据渲染到 HTML 页面中。  
`render_page()` 是统一渲染入口，它会自动注入：

- `request`
- 当前登录用户
- 闪现消息
- 区块链配置是否可用

这样可以减少各个路由中的重复代码。

### 3.2 用户输入处理

模块中包含多个通用数据处理函数，例如：

- `normalize_username()`：统一用户名格式
- `normalize_wallet_address()`：校验并标准化钱包地址
- `parse_collection()`：解析候选项列表或允许用户列表
- `parse_datetime_field()`：把页面输入的本地时间转换成 UTC 时间

这些函数的作用是让前端表单数据进入业务层前先完成一次清洗和校验。

### 3.3 投票状态判断

`poll_status_from_model()` 根据当前时间和投票的起止时间，将投票状态划分为：

- `pending`：未开始
- `active`：进行中
- `ended`：已结束

这个状态判断会直接影响：

- 页面显示什么按钮
- 投票能否编辑
- 结果能否公开
- 用户能否继续参与

### 3.4 投票可变更规则

`is_poll_mutable()` 明确规定：

- 只有未开始的投票才允许修改
- 且该投票还不能存在已同步的投票记录

这一设计可以避免链上投票已开始后，应用层继续修改投票内容，导致链上与链下信息不一致。

### 3.5 可参与投票过滤

`is_live_poll_available_to_user()` 的作用是筛选“当前用户现在真正还能参与的投票”。  
它会排除已结束投票，从而避免用户在“可参与列表”中点进去却只看到结果页。

### 3.6 链上同步校验

`verify_chain_sync_payload()` 是这个模块中非常关键的一段安全逻辑。  
它不会直接相信前端传回来的 `action_type`、`tx_hash` 和 `candidate_id`，而是会再次查询链上事件，确认：

- 用户的钱包是否绑定
- 交易哈希是否合法
- 链上记录的动作类型是否一致
- 候选项编号是否一致
- 链上返回的交易哈希是否与提交值匹配

这样做可以防止前端伪造同步数据。

## 4. 主要路由

### 4.1 认证与账户相关

- `/register`：注册页面与注册提交
- `/login`：登录页面与登录提交
- `/logout`：退出登录
- `/profile`：查看与更新绑定钱包地址

### 4.2 投票管理相关

- `/dashboard`：用户工作台
- `/polls/new`：新建投票
- `/polls/{poll_id}/edit`：编辑投票
- `/polls/{poll_id}/delete`：删除投票

### 4.3 投票展示与结果相关

- `/`：首页
- `/polls/{poll_id}`：投票详情页或结果页
- `/polls/{poll_id}/results`：结果页
- `/polls/{poll_id}/sync-chain-action`：同步链上交易回执

## 5. 与其他模块的关系

- 调用 `app/models.py` 中的 `User`、`Poll`、`PollVote`
- 调用 `app/database.py` 提供数据库连接与初始化
- 调用 `app/security.py` 完成密码哈希与验证
- 调用 `app/chain_service.py` 完成合约部署、链上状态读取和事件查询
- 向 `app/templates/` 提供页面渲染数据

## 6. 在系统中的价值

如果把整个项目看成一个分层系统，那么 `app/main.py` 就是“应用编排层”。  
它不负责真正上链，也不负责直接存储数据，但它负责把“用户操作”“数据库状态”“链上状态”连接成一个完整流程，是整个系统最核心的业务协调模块。

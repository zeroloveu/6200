# 测试与质量保障模块说明

## 1. 模块定位

本项目的测试主要分为两部分：

- Solidity/Hardhat 测试
- Python 后端单元测试

它们共同保证系统在“链上逻辑”和“链下业务逻辑”两个层面都具备可验证性。

## 2. Solidity 测试

目录位于 `test/`，包括：

- `unit/`：单元测试
- `integration/`：集成测试
- `performance/`：性能与 gas 测试
- `helpers/`：测试部署辅助函数

### 2.1 单元测试

`test/unit/SecureVoting.unit.test.js` 主要验证：

- 合约初始配置是否正确
- 只有拥有者可以注册选民
- 重复注册和零地址是否被拒绝
- 投票时间限制是否生效
- 一个地址是否只能参与一次
- 弃权逻辑是否正确
- 非法候选项和未注册地址是否被拒绝
- 状态转换是否正确

### 2.2 集成测试

`test/integration/SecureVoting.integration.test.js` 主要验证完整流程，包括：

- 多人登记
- 多人投票
- 弃权
- 投票结束后的结果读取
- 胜者计算
- 平票判断

### 2.3 性能测试

`test/performance/SecureVoting.performance.test.js` 主要评估：

- 批量注册 gas 使用量
- 单次投票 gas 使用量
- 弃权 gas 使用量

这有助于在报告中展示合约实现不仅正确，而且具备一定性能控制意识。

## 3. Python 测试

`app/tests/test_main.py` 主要验证：

- 投票何时允许编辑
- 已结束投票是否会错误出现在可参与列表中
- 链上同步是否以链上数据为准
- 候选项不匹配时是否拒绝同步
- 用户未绑定钱包时是否拒绝同步
- 链桥空输出或非法输出时是否能给出明确错误

## 4. 价值总结

测试模块的价值主要体现在三个方面：

- 证明关键功能是可验证的
- 降低修改代码后的回归风险
- 为课程报告提供“系统经过验证”的依据

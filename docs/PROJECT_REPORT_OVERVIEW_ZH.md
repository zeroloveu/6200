# Secure Voting 项目整体流程与功能说明

## 1. 项目概述

本项目是一个“链下管理 + 链上记票”的混合式电子投票系统。  
系统使用 FastAPI 构建 Web 应用，用于完成用户注册、登录、钱包绑定、投票创建和结果展示；同时使用 Solidity 编写智能合约，将真正的投票行为和计票结果记录在区块链上。

该架构的核心思想是：

- 将用户体验和业务管理放在传统 Web 应用中完成
- 将关键投票事实和最终结果放到链上，保证透明性和不可篡改性

因此，本项目既保留了网页系统易用、易管理的优点，也引入了区块链在可信记录方面的优势。

## 2. 项目功能

### 2.1 用户账户功能

系统支持用户注册、登录和退出登录。  
注册后，用户可以在个人资料页面绑定自己的区块链钱包地址。该地址是用户后续参与链上投票的唯一身份标识。

### 2.2 投票创建与发布

已登录用户可以创建投票，并设置：

- 投票主题
- 开始时间
- 结束时间
- 候选项列表
- 允许投票的用户名单

当用户提交创建请求后，后端会自动调用链桥脚本，把这次投票部署成一个独立的 `SecureVoting` 合约。  
也就是说，系统中的每一个投票实例，都会对应一个独立的链上合约地址。

### 2.3 白名单投票

投票并不是对所有注册用户开放，而是采用“允许名单”机制。  
只有在创建投票时被加入名单、且已经绑定钱包地址的用户，才有资格参与该投票。

这一机制增强了系统对投票资格的控制能力，更适合课程项目中“小范围治理、社团选举、班级投票”等场景。

### 2.4 链上投票与弃权

在投票开始后，允许名单中的用户可以通过浏览器钱包发起链上交易，完成：

- 对某个候选项投票
- 主动弃权

每个钱包地址只能参与一次，投票和弃权都算作一次有效参与。  
这一规则由智能合约强制执行，而不是由前端页面单独决定。

### 2.5 结果公开与核验

投票结束后，系统会展示链上读取的投票结果，包括：

- 每个候选项的票数
- 弃权数
- 已登记选民数
- 胜者或平票信息
- 合约地址
- 部署交易链接

任何人都可以通过合约地址或区块浏览器复核结果，因此结果具有较强的可验证性。

## 3. 系统整体架构

整个系统可以分为五层：

### 3.1 表现层

由 Jinja2 模板和 CSS 组成，负责用户界面展示和基础交互。

### 3.2 应用层

由 `app/main.py` 组成，负责路由处理、业务编排和页面渲染。

### 3.3 数据层

由 `app/models.py` 和 `app/database.py` 组成，负责本地数据库存储与对象映射。

### 3.4 链桥层

由 `app/chain_service.py` 和 `scripts/fastapi-chain.js` 组成，负责连接 Python 后端和区块链节点。

### 3.5 区块链可信层

由 `contracts/SecureVoting.sol` 构成，负责投票资格控制、链上记票和最终结果统计。

## 4. 业务流程

### 4.1 用户注册与绑定钱包流程

1. 用户访问注册页面并创建账户
2. 系统对密码进行哈希后存储
3. 用户登录后进入个人资料页面
4. 用户填写并绑定钱包地址
5. 系统校验钱包地址格式并保存

这一步完成后，用户才具备参与链上投票的基础条件。

### 4.2 投票创建与部署流程

1. 创建者进入“新建投票”页面
2. 填写主题、时间、候选项和允许用户名单
3. 后端校验表单内容是否合法
4. 后端根据允许用户名单查出对应用户的钱包地址
5. Python 后端调用 Node 链桥脚本
6. Node 脚本调用 Ethers.js 部署 `SecureVoting` 合约
7. 后端保存投票记录和合约地址
8. 页面跳转到投票详情页

这一流程实现了“从网页操作到链上部署”的自动化闭环。

### 4.3 用户投票流程

1. 被允许参与的用户进入投票详情页
2. 页面读取链上实时状态和候选项信息
3. 用户通过浏览器钱包发起投票交易
4. 区块链确认后，用户将交易同步回应用
5. 后端再次查询链上事件，核验交易是否真实
6. 核验成功后，把交易哈希和本地记录写入数据库

这里特别重要的一点是：  
应用不会盲目信任前端提交的数据，而是会再次查询链上事件，确认同步内容真实有效。

### 4.4 结果展示流程

1. 投票结束后，任何人都可以打开结果页
2. 后端调用链桥读取合约摘要
3. 页面展示链上统计结果
4. 用户也可以通过区块浏览器二次验证

因此结果页展示的不是应用自己计算的本地统计，而是来自链上合约的实时结果。

## 5. 安全与可信设计

### 5.1 账号安全

- 用户密码经过 `scrypt` 哈希后存储
- 不保存明文密码

### 5.2 投票资格控制

- 只有白名单钱包地址可以投票
- 只有绑定钱包的注册用户才能参与

### 5.3 一人一次

- 合约使用 `hasVoted` 映射限制重复投票
- 投票与弃权都只允许一次

### 5.4 时间窗口控制

- 只有在开始时间和结束时间之间才能投票
- 结束前无法公开最终结果

### 5.5 结果公开可验证

- 票数直接来自链上
- 合约地址与交易哈希可公开核验
- 结果不依赖本地手工统计

### 5.6 链上同步真实性校验

- 同步投票动作时，会重新读取链上事件
- 防止前端伪造候选项编号或交易哈希

## 6. 工程实现特点

本项目除了完成业务功能外，也体现了较完整的工程化设计：

- 使用 ORM 管理数据库模型
- 使用模板系统构建完整前端页面
- 使用 Hardhat 管理合约编译与测试
- 使用 Docker 实现容器化部署
- 使用单元测试和集成测试验证关键逻辑

这使得项目不仅是一个概念性原型，也具备一定的可复现和可部署能力。

## 7. 项目优势

- 兼顾 Web 易用性和区块链可信性
- 支持钱包绑定和白名单控制
- 每个投票对应独立合约，结构清晰
- 结果透明，便于外部核验
- 适合作为课程项目、毕业设计原型和课堂展示案例

## 8. 项目局限性

虽然本项目具备较完整的功能，但仍然属于教学和演示导向系统，存在以下局限：

- 默认使用 SQLite，更适合单机演示
- 依赖外部 RPC 服务，可能受限流或超时影响
- 未实现更复杂的身份认证机制
- 不具备正式政治选举所需的法律、合规和隐私保障体系

因此，本项目更适合作为“可信投票系统原型”，而不是直接用于高风险正式选举。

## 9. 总结

总体来看，本项目实现了一套完整的混合式投票系统：

- 用户在网页中注册、登录和管理投票
- 投票资格通过白名单与钱包绑定控制
- 实际投票动作写入区块链
- 最终结果从链上读取并公开展示

该方案清晰展示了区块链技术在“防篡改、可审计、可公开验证”方面的优势，也体现了传统 Web 应用在管理流程和用户体验方面的重要价值。  
对于课程报告而言，这个项目既有明确的系统架构，也有完整的业务流程和较好的工程实现基础。

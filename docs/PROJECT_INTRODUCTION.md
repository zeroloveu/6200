# Secure Voting 项目介绍文档

## 1. 项目概述

`Secure Voting` 是一个基于以太坊智能合约的课程型电子投票项目，目标是在较小的实现规模内展示区块链在投票场景中的几个核心价值：

- 数据公开可审计
- 投票结果不可随意篡改
- 选民资格控制可编程
- 投票时间窗口可链上强制执行

该项目当前采用 `Hardhat + Solidity + Ethers.js` 技术栈实现，并已完成本地测试、Sepolia 测试网部署以及 Etherscan 源码验证。它适合用于课程作业、毕业设计原型、区块链实验报告和教学演示，不适合直接作为正式政治选举或高风险治理系统投入生产。

## 2. 项目目标

本项目要解决的不是“如何构建一个完全匿名、可扩展、可监管的真实选举系统”，而是“如何用一套简洁、可验证、可演示的智能合约流程，展示链上投票的基本机制”。

围绕这个目标，项目重点实现了以下能力：

- 管理员控制选民资格，只有白名单地址可以投票
- 每个地址只能投一次票，避免重复投票
- 投票只能在设定的开始时间和结束时间之间进行
- 候选人票数保存在链上，任何人都可以验证
- 选举结束后可以读取完整结果并确定获胜者
- 合约实现尽量保持简洁，便于教学和答辩说明

## 4. 系统设计思路

### 4.1 参与角色

系统中有三类核心参与者：

- `Owner / 管理员`
  负责部署合约，初始化候选人，设置投票时间，并在需要时注册选民
- `Voter / 选民`
  必须先进入白名单，才能在投票开放期间对候选人投票
- `Observer / 观察者`
  即使不参与投票，也可以读取链上状态、候选人信息和最终结果

### 4.2 链上业务流程

项目的标准流程如下：

1. 管理员部署合约
2. 部署时写入选举标题、候选人名单、开始时间、结束时间和初始选民列表
3. 若后续有新选民，管理员可调用注册接口追加白名单
4. 当区块时间进入投票窗口后，已注册选民可以调用 `vote(candidateId)` 投票
5. 每个地址只能投一次票
6. 投票结束后，任何人都可以读取结果并查询赢家

该流程的特点是“规则在链上执行”，而不是依赖中心化数据库去判定某个选民是否重复投票、是否在正确时间投票。

## 5. 智能合约设计说明

### 5.1 核心状态变量

合约中保存了以下关键数据：

- `owner`
  合约管理员地址，只有它能注册选民
- `electionTitle`
  选举标题
- `startTime` / `endTime`
  投票开始与结束时间戳
- `totalVotes`
  已投出的总票数
- `totalRegisteredVoters`
  已注册的选民总数
- `isRegisteredVoter`
  记录某地址是否具备投票资格
- `hasVoted`
  记录某地址是否已经投票
- `candidates`
  候选人编号到候选人信息的映射

### 5.2 候选人结构

每个候选人由如下结构表示：

- `name`
  候选人名称
- `voteCount`
  候选人当前获得的票数

### 5.3 构造函数职责

部署合约时，构造函数会一次性完成以下工作：

- 校验时间区间是否合法
- 校验候选人数量是否在允许范围内
- 保存管理员地址、选举标题和时间窗口
- 初始化候选人列表
- 注册初始选民

这意味着一次部署对应一场确定的选举，候选人和时间参数在部署后不会被重新配置。

## 6. 主要功能接口

### 6.1 选民注册

- `registerVoter(address voter)`
  由管理员注册单个选民
- `batchRegisterVoters(address[] calldata voters)`
  由管理员批量注册选民

约束条件：

- 只有 `owner` 能调用
- 不能注册零地址
- 不能重复注册
- 单次批量注册数量受上限控制，避免 gas 过高

### 6.2 投票

- `vote(uint256 candidateId)`

调用前提：

- 当前时间必须在投票窗口内
- 调用者必须已经被注册
- 调用者尚未投票
- `candidateId` 必须有效

调用成功后会发生：

- `hasVoted[msg.sender] = true`
- `totalVotes += 1`
- 对应候选人票数 `+1`
- 触发 `VoteCast` 事件

### 6.3 状态查询

- `candidateCount()`
  返回候选人总数
- `getCandidate(uint256 candidateId)`
  查询指定候选人的名称和得票数
- `getElectionStatus()`
  返回 `PENDING`、`ACTIVE` 或 `ENDED`

### 6.4 结果查询

- `getAllResults()`
  选举结束后返回所有候选人的名称及票数
- `getWinner()`
  选举结束后返回获胜者编号、名称、票数以及是否平票

该设计使得投票过程和结果查询严格分阶段执行，避免在投票尚未结束时过早读取最终结果。

## 7. 权限控制与安全设计

本项目虽然不是生产级选举系统，但已经纳入了若干基础安全设计。

### 7.1 权限控制

- 通过 `onlyOwner` 修饰器限制注册接口
- 普通选民不能私自把自己加入白名单
- 普通地址不能修改候选人列表或选举时间

### 7.2 时间窗口控制

- `duringElection`
  用于限制只有在 `startTime <= now <= endTime` 时才能投票
- `afterElection`
  用于限制只有在投票结束后才能读取完整结果和赢家

### 7.3 输入校验

- 候选人至少 2 名，且不能超过上限
- 候选人名称不能为空
- 结束时间必须晚于开始时间
- 不能注册零地址
- 不能向不存在的候选人投票

### 7.4 防重复投票

- 使用 `hasVoted` 记录地址是否已经投票
- 一旦投票成功，该地址将无法再次投票

### 7.5 Gas 与实现复杂度控制

- 使用自定义错误代替长字符串回退，减少 gas 开销
- 批量注册设置上限，避免单笔交易过重
- 投票流程中不包含外部调用，降低重入相关风险

## 8. 测试体系与质量验证

项目已经包含较完整的自动化测试，覆盖功能正确性、流程连通性和基础 gas 表现。

### 8.1 单元测试

单元测试覆盖了以下场景：

- 部署后的初始配置是否正确
- 是否只有管理员可以注册选民
- 是否拒绝零地址和重复注册
- 是否阻止开始前投票
- 是否允许已注册选民正常投票
- 是否阻止未注册地址投票
- 是否拒绝无效候选人编号
- 是否阻止结束后投票
- 状态是否正确在 `PENDING / ACTIVE / ENDED` 之间切换

### 8.2 集成测试

集成测试验证了完整业务链路：

- 批量注册额外选民
- 多地址参与投票
- 结束前禁止读取最终结果
- 结束后正确返回所有候选人的票数
- 正确识别赢家
- 正确识别平票情况

### 8.3 性能测试

项目还加入了基础 gas 约束：

- 批量注册总 gas 应低于 `900000`
- 批量注册平均每个选民 gas 应低于 `90000`
- 单次投票平均 gas 应低于 `120000`

这些指标不是绝对性能上限，但足以支持课程演示中对“链上操作成本”的讨论。

```bash

```

```bash
npm run deploy:sepolia
```

部署脚本已经支持：

- 根据环境变量初始化选举参数
- 自动等待确认
- 自动提交 Etherscan 验证

```bash
npm run status:sepolia
npm run register:sepolia
npm run vote:sepolia
```

相关环境变量包括：

- `CONTRACT_ADDRESS`
- `VOTER_ADDRESS`

## 12. 项目局限性

该项目仍然存在明显边界，不能等同于真实世界选举基础设施。

主要局限包括：

- 不具备匿名性，地址和投票行为可被链上关联
- 不包含链下身份认证，白名单真实性依赖管理员
- 不支持多轮投票、委托投票、加密投票等高级机制
- 不支持候选人变更或重新配置选举参数
- 投票结果完全公开，不适合隐私敏感场景
- 管理员私钥安全仍然是核心信任点

## 13. 适用场景

该项目适合以下场景：

- 区块链课程作业
- 智能合约实验
- 投票系统原型展示
- 毕业设计中的功能模块演示
- 展示链上透明性与不可篡改性的教学案例

不适合以下场景：

- 正式政治选举
- 涉及严格隐私保护的投票
- 需要大规模并发选民支持的生产系统
- 需要法律合规审计与复杂治理流程的真实机构场景

## 14. 总结

`Secure Voting` 项目展示了一个典型的智能合约投票系统如何将“选民资格控制、单次投票约束、时间窗口限制、结果公开可验证”这些规则固化到链上执行逻辑中。它的优势不在于功能极其复杂，而在于实现边界清晰、结构简洁、便于验证和展示。

对于课程或项目答辩而言，该项目的价值主要体现在三点：

- 能够清楚说明区块链如何提升投票结果的透明性和不可篡改性
- 能够展示智能合约如何替代部分中心化业务规则
- 能够以较低理解门槛完成从开发、测试、部署到链上交互的完整闭环

如果后续要继续扩展，这个项目可以进一步演化为：

- 前端投票页面
- 钱包连接与身份管理模块
- 更强的隐私保护机制
- 多阶段选举与治理系统
- 更完善的运维与监控方案
